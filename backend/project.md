# Описание

RENTX - сервис аренды вещей, в котором расчеты, депозиты и подтверждение сделок завязаны на крипто-токены в сети Solana.

Платформа позволяет владельцам публиковать вещи в аренду, арендаторам бронировать их на определенный срок, а backend-части хранить каталог, правила аренды, статусы сделок, отзывы и интеграцию с on-chain платежами.

# Цель MVP

- Дать пользователю возможность подключить Solana-кошелек и создать профиль.
- Разместить объявление о сдаче вещи в аренду.
- Найти объявление по фильтрам и создать заявку на аренду.
- Зафиксировать стоимость аренды, депозит и срок аренды.
- Подтвердить факт передачи и возврата вещи.
- Сохранить результат сделки и оставить отзыв.

# Основной сценарий

1. Владелец регистрируется через Solana-кошелек.
2. Владелец создает объявление, добавляет фото, описание, цену, депозит и доступность.
3. Арендатор находит объявление и отправляет запрос на аренду.
4. Система создает сущность аренды со статусом ожидания подтверждения.
5. После подтверждения сделки фиксируется платежный контекст: токен, сумма аренды, сумма депозита, сетевые идентификаторы транзакций.
6. Во время аренды backend хранит текущее состояние сделки.
7. После возврата вещи аренда закрывается, депозит либо возвращается, либо удерживается частично/полностью по правилам платформы.
8. Обе стороны могут оставить отзывы друг о друге и о предмете аренды.

# Сущности

## Users

Пользователи платформы.

Основные поля:

- `id`
- `walletAddress` - основной адрес кошелька Solana
- `username`
- `displayName`
- `avatarUrl`
- `bio`
- `rating`
- `reviewsCount`
- `isVerified`
- `createdAt`
- `updatedAt`

Роль в системе:

- Пользователь может быть владельцем вещи.
- Пользователь может быть арендатором.
- Один пользователь может совмещать обе роли.

Связи:

- `User -> Posts` как автор объявлений
- `User -> Rents` как арендатор
- `User -> Reviews` как автор отзыва
- `User -> Reviews` как получатель отзыва

## Posts

Объявления о сдаче вещей в аренду.

Основные поля:

- `id`
- `ownerId`
- `title`
- `description`
- `category`
- `pricePerDay`
- `depositAmount`
- `currencyMint` - mint токена в Solana
- `location`
- `status` - `draft | active | paused | rented | archived`
- `availableFrom`
- `availableTo`
- `createdAt`
- `updatedAt`

Дополнительно:

- У одного объявления может быть несколько фотографий.
- У объявления могут быть произвольные атрибуты в зависимости от категории.
- После публикации объявление участвует в поиске и бронировании.

## Posts Attributes

Гибкий набор характеристик объявления.

Назначение:

- Не хранить все свойства вещи в одной широкой таблице.
- Поддерживать разные категории: электроника, транспорт, инструменты, одежда и т.д.

Основные поля:

- `id`
- `postId`
- `key`
- `value`
- `type` - `string | number | boolean | json`

Примеры атрибутов:

- `brand=Canon`
- `condition=used`
- `power=1200`
- `size=M`
- `serialNumber=...`

## Rents

Сделки аренды между владельцем и арендатором.

Основные поля:

- `id`
- `postId`
- `ownerId`
- `renterId`
- `startDate`
- `endDate`
- `daysCount`
- `pricePerDay`
- `rentAmount`
- `depositAmount`
- `platformFeeAmount`
- `totalAmount`
- `currencyMint`
- `paymentTxSignature`
- `depositTxSignature`
- `returnTxSignature`
- `status`
- `cancelReason`
- `createdAt`
- `updatedAt`

Возможные статусы:

- `pending` - заявка создана, ожидает подтверждения
- `approved` - подтверждена владельцем
- `rejected` - отклонена владельцем
- `paid` - платеж подтвержден
- `active` - предмет передан арендатору
- `completed` - аренда завершена успешно
- `cancelled` - аренда отменена
- `disputed` - открыт спор

Бизнес-правила:

- Стоимость аренды фиксируется в момент подтверждения заявки.
- Депозит хранится отдельно от стоимости аренды.
- Для одной вещи не должно быть пересекающихся активных аренд на одинаковый период.
- On-chain транзакции должны быть связаны с записью аренды в backend.

## Reviews

Отзывы после завершения аренды.

Основные поля:

- `id`
- `rentId`
- `authorId`
- `targetUserId`
- `postId`
- `rating`
- `comment`
- `createdAt`

Правила:

- Отзыв можно оставить только по завершенной аренде.
- Один пользователь оставляет не более одного отзыва на одну сторону в рамках одной аренды.
- Рейтинг пользователя пересчитывается на основе подтвержденных отзывов.

# Дополнительные сущности, которые понадобятся

## Post Images

Файлы изображений объявления, хранящиеся в Minio.

Поля:

- `id`
- `postId`
- `objectKey`
- `url`
- `sortOrder`

## Wallet Sessions / Auth Nonces

Сущность для аутентификации через подпись кошельком.

Поля:

- `id`
- `walletAddress`
- `nonce`
- `expiresAt`
- `usedAt`

## Rent Events

Журнал изменений аренды.

Поля:

- `id`
- `rentId`
- `type`
- `payload`
- `createdAt`

Назначение:

- хранить историю переходов статусов
- фиксировать технические события интеграции с Solana
- упрощать аудит и разбор спорных ситуаций

# Инструменты

- PostgreSQL - основное хранилище данных
- TypeORM - ORM для работы с сущностями и миграциями
- Minio - хранение изображений и файлов объявлений
- Docker - локальная разработка и контейнеризация сервисов
- NestJS - backend framework для API и бизнес-логики

# Интеграция с Solana

## Что хранить в backend

- адрес кошелька пользователя
- mint токена для расчетов
- суммы аренды и депозита
- подписи транзакций (`txSignature`)
- внутренние статусы синхронизации on-chain/off-chain

## Что не стоит хранить как источник истины

- фактический баланс кошелька
- подтверждение транзакции без верификации через RPC
- приватные ключи пользователей

## Возможные направления интеграции

- авторизация по подписи сообщения кошельком
- оплата аренды в SPL token
- депозит через escrow-механику
- чтение статуса транзакции через Solana RPC

# Базовые API-модули backend

- `auth` - вход по подписи кошелька
- `users` - профиль пользователя
- `posts` - объявления
- `files` - загрузка изображений в Minio
- `rents` - сделки аренды
- `reviews` - отзывы
- `payments` - проверка и привязка транзакций Solana

# Риски и ограничения

- Нужно отдельно продумать escrow-логику депозита: полностью on-chain или гибридно.
- Нужна защита от двойного бронирования одного предмета.
- Нужно определить, в какой валюте считать цену: SOL, стейблкоин или кастомный SPL token.
- Нужно продумать dispute flow для спорных возвратов и удержания депозита.
